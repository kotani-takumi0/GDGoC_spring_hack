# リアサポ — プロダクト設計書

## GDGoC Japan Hackathon 2026 提出用

---

## 1. プロダクト概要

### 一言で言うと

**AI時代の新しい"Hello World"体験を作る学習支援プロダクト**

### テーゼ

> 従来のHello World：「コードを書いて動かすこと」が最初の一歩
>
> AI時代のHello World：**「自分が何を理解すべきかを特定すること」が最初の一歩**

### 解いている問題

AIコーディングツール（Copilot、Claude Code、Cursorなど）の普及により、初学者〜中級者が「AIにコードを書かせて動いたけど、何も理解していない」という状態に陥る問題（output ≠ understanding問題）。

根本原因は、学習者が**「自分が何を理解すべきか」をそもそも知らないまま**AIにコードを書かせていること。理解の地図（ロードマップ）がないから、「分からない」の自覚すら持てない。

### 問題の深刻さ（データ）

- 2025年時点で書かれるコードの約41%がAI生成、2026年後半には50%超と予測
- 大学生のAI利用率は2023年の36.8%から2025年には91.7%に急増（ArXiv 2025.11, 10,000件の学生-AIダイアログ分析）
- 学生はAI提供のソリューションを「正当化せず受け入れる」傾向が確認されている
- AI生成コードの約45%にセキュリティ脆弱性が含まれるとの研究結果
- CACM（2024-25）: AI時代のCS教育の中心スキルは「コードを書く」→「コードを理解・評価する」にシフトすべきと提言

### リアサポのアプローチ

**AIがコードを書く「前」に、理解のロードマップを手にする。**

学習者がAIにコード生成を依頼する前に、「このタスクを理解するために必要な概念」を依存関係付きのロードマップとして提示する。AIがコードを生成した後、学習者は各概念について自分の言葉で説明することで、理解度を確認する。

システムは**地図を渡すだけ**。歩くのは学習者自身。

## 2. Brand New "Hello World." へのアンサー

テーマ「Brand New "Hello World."」は、AI時代に「最初の一歩」の意味が変わったことを問いかけている。

- **昔のHello World**：コードを書く → 動く → 少しずつ理解する（入口は「書く」）
- **今のHello World**：AIが書く → 動く → でも理解していない（「書く」が代替された）
- **リアサポのHello World**：何を理解すべきか知る → 学ぶ → AIが書く → 答え合わせする（入口は「理解の地図を手にする」）

初めてリアサポでロードマップを開いて「自分が何を理解すべきか」を知った瞬間が、AI時代の新しい最初の一歩。

### この一歩の先にあるもの

現実のAIプログラミングでは、優れたエンジニアはAIにこう指示する：

> 「既存の関数使えるよね。そこ共通化して。bottom sheetは○○使って。自前実装にするかライブラリにするか迷ってるから候補を比較して。セキュリティルールあるから先に確認お願い。」

これができるのは、概念の存在を知り、依存関係を理解し、その上で技術判断を下せるから。リアサポが提供する「概念の地図を持つ」体験は、この能力の**スキップできない一歩目**にあたる。

| 段階 | 能力 | リアサポの対応 |
|------|------|--------------|
| 1. 概念の存在を知る | 「状態管理」という概念があると知っている | Step 2（ロードマップ生成） |
| 2. 依存関係を理解する | 「状態管理の前にデータ構造を理解すべき」と分かる | Step 2（グラフの依存関係）→ Step 4〜5（コードで確認・言語化） |
| 3. 技術判断を下せる | 「このケースはuseStateで十分」と選べる | 将来構想（選択肢・判断基準のレイヤー追加） |

**リアサポは段階1→2を提供し、段階3に至る道を開く。**

## 3. コア体験フロー

```
[MCPクライアント側 or Webアプリ]
  [Step 1] 学習者がシナリオと経験レベルを選択
           シナリオ：選択式（Todoアプリ等、3〜5個の固定シナリオ）
           経験レベル：選択式（完全初心者 / Python経験あり / 他言語経験あり 等）

      → Webアプリに遷移（ロードマップの確認）

[Webアプリ]
  [Step 2] 概念ロードマップをビジュアル表示
           ノード構造・依存関係 → シナリオごとに事前定義（静的JSON）
           各ノードの一言説明 → Gemini APIがユーザーの経験レベルに合わせて動的生成
           学習者が構造を確認 →「OK、実装して」

[MCPクライアント側]
  [Step 3] AIがコード生成開始 → 即座にWebアプリに遷移

[Webアプリ]
  [Step 4] 実装中にロードマップを再確認
           完了後、ロードマップとコードが並んで表示される
           概念とコードが線で結ばれる
      ↓
  [Step 5] 学習者がコードを見ながら理解度を評価
           「この部分が何をしているか、説明できますか？」
```

### 2つの入口（Webアプリ + MCP）

**メイン：Webアプリ完結**
- 初学者向け。URLを開くだけで全ステップを体験できる
- Webアプリ内でコード生成もGemini APIで完結させる
- 審査デモはこちらを使用

**サブ：MCPサーバー対応（⚠️ ハッカソン版ではスコープ外）**
- Cursor/Claude Code等を既に使っている初中級者向け
- 普段の開発環境から離れずに同じ体験ができる
- Step 1はMCPクライアント側、Step 2以降はWebアプリに遷移
- MCPサーバーはWebアプリと同じAPIを叩く（実装コスト小）
- **ハッカソン版ではWebアプリ完結を最優先とし、MCPサーバーは実装しない。将来構想として記載を残す**

> 「初学者はWebアプリで体験できる。成長してCursorやClaude Codeを使うようになったら、MCPで普段の開発環境から同じ体験ができる。成長に合わせて入口が変わる設計。」（※ハッカソン版ではWebアプリのみ実装）

### ハッカソンルールへの準拠

#### 開発開始前に許可されている作業
- 開発環境の整備（ソフトウェア、エディタ、言語ランタイム等のインストール）
- 技術検証を目的とした概念実証（PoC）
- デザインツールなどを用いたUIデザインの作成
- 設計書・仕様書の作成（本ドキュメント含む）

#### 開発開始前に禁止されている作業
- プロダクトに関するあらゆるソースコードの記述
- インフラ構築コード（Terraform等のIaC）の作成
- CI/CDパイプラインの構築・設定
- フレームワーク等の初期化コマンド（ボイラープレートの生成）の実行

#### 提出物に関する注意
- 開発可能期間外に記述・生成されたコードを提出物に含めることは禁止
- PoCで作成したコードを本番リポジトリにコピー＆ペースト・流用することは禁止
- 開発開始後に必ずゼロから記述・生成すること

### スコープの方針（ハッカソン版）

**シナリオ固定 × 体験の質に全振り**する方針を採用。

- プロダクトの定義：「AI時代のプログラミング学習で、最初の一歩がどう変わるかを示すプロトタイプ」
- 汎用学習ツールではなく、**新しいHello World体験のデモンストレーション**として位置づける
- シナリオを固定することで体験の質を100%コントロールし、審査員がプロダクトの思想に集中できるようにする

#### シナリオ固定でも技術パイプラインは本物を通す

固定シナリオ ≠ ハードコード。各ステップで実際にAI（Gemini API）を呼び出す。固定シナリオはパイプラインの品質を担保するための**チューニング済みの入力**であり、裏の技術は本物。

#### なぜStep 2でGeminiを使うか（パーソナライズ）

ロードマップの構造（ノードと依存関係）は固定で十分。しかし**説明文はユーザーの経験レベルに合わせて変わるべき**。同じ「状態管理」ノードでも：

- 完全初心者 → 「データを覚えておく仕組み」
- Python経験者がReact初挑戦 → 「Pythonの変数に近いが、UIと自動連動する」

これにより、Gemini APIを使う**明確な理由（パーソナライズ）**が生まれ、技術点を確保できる。

### ポイント

- **Step 2 が核心**：地図を見せるだけで「あ、5つも知るべきことがあるのか」「永続化って何だろう」という気づきが発生する。これがメタ認知の発火点であり、評価や入力を求める必要はない
- **Step 4 が答え合わせ**：実装前に見た概念が、コードの具体に結びつく瞬間。「状態管理ってuseStateのことだったのか」と3層で繋がる
- **評価はコード生成後**：抽象的な概念に対して「分かる？」と聞いても初学者は正確に答えられない。具体的なコードがあって初めて「この行が何をしているか説明できるか？」という問いが成立する
- **学習の主体は学習者**：システムは地図を見せ、コードとの対応を示すだけ。「ここが分からないよ」と教えるのではなく、学習者が「あ、ここ分かってなかった」と自分で気づく体験を設計する

### なぜ実装前に自己評価を求めないか

旧設計ではStep 2の直後に自己評価（分かる/怪しい/分からない）を求めていたが、削除した。理由：

- リアサポが解いている問題は「自分が何を知らないか分からない」。その人に実装前の自己評価を求めても正確な答えは出ない
- 初学者に「状態管理、分かる？」と聞いても、コードを見たことがないから判断のしようがない
- 地図を**見せること自体**がメタ認知を発火させる。🟢🟡🔴を付ける行為が本質ではない
- 「地図を渡すだけ。歩くのは学習者自身」の思想に最も忠実な形

### 学術的裏付け

この体験設計は、以下の教育研究の知見と合致する：

- **メタ認知キャリブレーション**（CHI 2025）：自分の理解度と実際の理解度のギャップを認識する能力の向上が、学習成果を改善することが確認された。Step 5の理解度評価がこのキャリブレーション行為にあたる
- **PRIMM手法**（Predict-Run-Investigate-Modify-Make）：英国の13校・493人の研究で「予測→確認サイクル」が学習達成度を向上させることを実証。Step 2（概念を知る）→ Step 4（コードで確認）のフローはPRIMMの構造と一致
- **メタ認知スキャフォールディング**（ArXiv 2025.11）：「AIが答えを提供する前に、学生に考えや仮説を表明させるプロンプトをシステム設計に組み込む」べきとの提言。Step 2のロードマップ提示が「先に概念の枠組みを持たせる」役割を果たす
- **能力の幻想への対策**：コードを「生成できる」ことを「理解できる」と誤認するメタ認知バイアス（Dunning-Kruger効果に類似）に対し、Step 5で具体的なコードへの理解を問うことで、この錯覚を構造的に破る

### UI・インタラクション設計

#### 概念ノードの設計

各ノードは**技術用語（タイトル）+ 平易な一言説明（サブテキスト）** で構成する。技術用語は学ぶべき対象なので隠さない。ただし初学者が意味を掴めるよう一言説明を併記する。

```
[状態管理]
 データを覚えておく仕組み

    ↓

[リスト表示]
 一覧を並べて表示する

    ↓

[イベントハンドリング]
 ボタンを押したときの動作
```

ノード数は1タスクあたり**5〜8個**。一画面に収まり、初学者が「地図」として認識できる上限。

#### 概念の粒度の段階変化

概念ノードは実装の前後で抽象度が変わる：

- **Step 2（実装前）**：概念レベル。「状態管理」「リスト表示」など、コードを知らなくても理解できる粒度。色はなし（まだ評価していない）
- **Step 4（実装後）**：同じノードにAPI/構文レベルが追加される。概念がコードの具体に結びつく

```
【Step 2：実装前（見るだけ）】
  [状態管理]
   データを覚えておく仕組み

【Step 4：実装後（コードと結びつく）】
  [状態管理]
   データを覚えておく仕組み
   → const [todos, setTodos] = useState([])
```

学習者は「状態管理 = useState = データを覚えておく仕組み」と3層で理解が繋がる。

#### 画面構成：ロードマップとコードを並べて線で繋ぐ

Step 4（コード生成後）では、ロードマップとコードを左右に並べ、対応箇所を線で接続する。ロードマップの依存関係（矢印）も維持されるため、学習順序の情報が失われない。

```
┌─ ロードマップ ──────────┐  ┌─ コード ───────────────────┐
│                          │  │                              │
│  [コンポーネント設計] ────┼──▶ function App() { ... }      │
│   画面を部品に分ける      │  │                              │
│        ↓                 │  │                              │
│  [状態管理] ─────────────┼──▶ const [todos] = useState()   │
│   データを覚えておく仕組み │  │                              │
│        ↓                 │  │                              │
│  [リスト表示] ───────────┼──▶ todos.map(t => ...)          │
│   一覧を並べて表示する    │  │                              │
│        ↓                 │  │                              │
│  [イベントハンドリング] ──┼──▶ onClick={handleAdd}          │
│   ボタンを押したときの動作 │  │                              │
│        ↓                 │  │                              │
│  [永続化] ───────────────┼──▶ localStorage.setItem(...)    │
│   閉じても消えないように   │  │                              │
│                          │  │                              │
└──────────────────────────┘  └──────────────────────────────┘
```

#### 理解度評価のインタラクション（Step 5）

コード生成後に、各概念ノードに対して**学習者が自分の言葉で説明する**。具体的なコードが目の前にあるため、「この部分が何をしているか、自分の言葉で説明してください」という問いが成立する。

学習者のテキスト回答をGemini APIが判定し、ノードに色が付く：
- 🟢 **説明できている** — 概念とコードの対応を正しく言語化できた
- 🟡 **部分的に理解** — 方向性は合っているが、説明に曖昧さがある
- 🔴 **理解できていない** — 説明が的外れ、または説明できなかった

色が付くことで、ロードマップが「自分の理解状態の地図」に変わる。

#### なぜボタン自己申告ではなくテキスト回答か

このプロダクトの目的は自動化でも効率化でもなく、**学習**である。あえてユーザーに負荷をかける設計を意図的に選択している。

- ボタンで🟢🟡🔴を選ぶだけでは「歩いていない」。自分の言葉で説明する行為そのものが学習
- 「システムは地図を渡すだけ。歩くのは学習者自身」の思想に最も忠実な形
- AIは学習者の理解を**代替するのではなく、判定する側**に回る
- 教育研究で実証されている「説明生成効果（generation effect）」：情報を自分の言葉で再構成すると記憶定着率が向上する

## 4. 技術アーキテクチャ

### 全体パイプライン

```
Step 1（シナリオ・経験レベル選択）
  学習者が固定シナリオ（3〜5個）と経験レベルを選択

Step 2（実装前：概念ロードマップ生成）
  ① 静的JSON：シナリオに対応する概念ノード・依存関係を取得（事前定義）
  ② Gemini API：経験レベルに基づき、各ノードの一言説明をパーソナライズ生成

Step 3（コード生成）
  ③ Webアプリ版：Gemini APIがコード生成
     MCP版：MCPクライアント側のAI（Cursor/Claude Code等）がコード生成
  → 即座にWebアプリに遷移

Step 4（実装後：概念↔コードのマッピング）
  ④ Gemini API：コード全体を渡し、各概念ノードに対応する箇所を関数・ブロック単位で返させる
     ※ AST解析は不要。初学者向けにはざっくり対応が分かれば十分。Gemma不要

Step 5（理解度評価）
  ⑤ 学習者が各概念ノードについて「この部分が何をしているか」を自分の言葉でテキスト回答
  ⑥ Gemini API：学習者の回答を判定し、理解度（🟢🟡🔴）を返す
```

※ Mermaid形式の図は `pipeline_diagram.md` を参照

### Google技術の活用

#### Gemini API — プロダクトの心臓部

リアサポの全ステップでGemini APIが中核を担う：

- **Step 2（パーソナライズ）**：事前定義された概念ノード構造に対し、ユーザーの経験レベルに合わせた一言説明を動的生成（構造化出力モード活用）
- **Step 3（コード生成）**：Webアプリ版ではGemini APIがシナリオに基づいてコードを生成
- **Step 4（マッピング）**：生成コードと概念ノードの対応を関数・ブロック単位で判定。「状態管理 = この辺りのuseState」のマッピングを生成
- **Step 5（理解度判定）**：学習者のテキスト回答を判定し、理解度（🟢🟡🔴）を返す

#### MCPサーバー — 開発環境との統合（⚠️ ハッカソン版ではスコープ外）

- リアサポをMCPサーバーとして公開し、Cursor/Claude Code等から接続可能にする
- 初学者はWebアプリ、成長した学習者はMCPと、成長に合わせて入口が変わる設計
- MCPサーバーはWebアプリと同じAPIを叩くため、実装コストは小さい
- **ハッカソン版ではWebアプリ完結を優先し、MCPサーバーは未実装。将来的にAPI設計を共通化しているため、後から追加可能**

#### 概念グラフDB — 事前定義の静的JSON

- シナリオごとに概念ノード（5〜8個）と依存関係を事前定義
- 構造の安定性を最優先。動的生成によるブレを排除する設計判断
- シナリオ数は3〜5個（ハッカソン版）。共通ノード（状態管理等）を意図的に配置し、グラフとしての再利用性を示す
- 各シナリオの具体的なノード定義・経験レベル定義は `scenario_definitions.md` を参照

#### 技術スタック（確定）

| レイヤー | 技術 | 役割 |
|---------|------|------|
| フロントエンド | React + Next.js | Webアプリ本体 |
| グラフ描画 | React Flow | 概念ロードマップの描画・線で結ぶUI |
| バックエンド | Next.js API Routes | Gemini API呼び出し等のサーバー処理 |
| インフラ | Cloud Run | Next.jsをDockerでデプロイ。Google Cloud上で完結 |
| AI | Gemini API | パーソナライズ・コード生成・マッピング・判定 |

フロント・バック・APIをNext.js 1プロジェクトに集約し、Cloud Runにデプロイするシンプル構成。

## 5. 市場ポジショニングと差別化

### 市場の空白

コードを構造・依存関係として可視化するツールは多数存在する（Windsurf Codemaps, CodeSee, Sourcegraph, CodeViz等）。しかし、それらは全て**プロ開発者・チーム向け**であり、**コードレベル**（ファイル・関数の依存関係）の可視化に留まる。

一方、初学者向けの学習ツール（Python Tutor, Khanmigo, Codecademy等）は**実行の可視化**や**インタラクティブ演習**に特化しており、「概念の依存関係」を構造化して見せるものは存在しない。

**リアサポが狙うポジション — 「初学者向け × 概念レベルの可視化 × AI生成コードの理解支援」 — に直接競合はいない。**

### 既存ツールとの根本的な違い

| 軸 | 既存ツール（Codemaps等） | リアサポ |
|----|------------------------|---------|
| 可視化の対象 | コード構造（ファイル・関数） | **概念の依存関係（知識グラフ）** |
| タイミング | コードが存在する「後」 | **AIに書かせる「前」** |
| ユーザーの役割 | 受動的に見る・質問する | **能動的に自己評価する** |
| ターゲット | プロ開発者・チーム | **AI時代の初学者** |
| 目的 | 開発効率の最大化 | **理解の質の保証** |

### 最も近い競合：Windsurf Codemaps

Codemapsは「Understand Code Before You Vibe It」を掲げ、AI生成コードの理解という同じ問題意識を持つ。ただし：

- Codemapsは**既存コードの地図を作る（after）**。リアサポは**これから書くコードの概念ロードマップを作る（before）**
- Codemapsのユーザーは既にコードが書ける人。リアサポのユーザーはまだ書けない人
- Codemapsはコード構造を可視化。リアサポは**学習概念の依存関係**を可視化

### 学習ツールとの違い

| ツール | アプローチ | リアサポとの差 |
|--------|-----------|--------------|
| Python Tutor | 実行ステップの可視化 | 実行レベル、概念の依存関係は扱わない |
| Khanmigo | ソクラテス式問答で誘導 | AIが教える。リアサポは学習者自身が気づく |
| roadmap.sh | キャリア単位の長期ロードマップ | 月・年単位。リアサポは「今このタスク」の分単位 |
| Codecademy / freeCodeCamp | インタラクティブ演習 | 体系的カリキュラム。リアサポはタスク起点のオンデマンド |

## 6. 審査基準との対応

### 1. テーマ性

- **テーマとの合致**：「Brand New "Hello World."」に対し、「AI時代の最初の一歩は、何を理解すべきかを特定すること」という明確なアンサー
- **ストーリー**：チームメンバー自身がAI開発初心者 → 「何を理解すべきか分からない」を身をもって体験 → その原体験からプロダクトが生まれた
- **Brand New度**：従来の学習ツール（Duolingo的な適応型学習）は「システムが教える」。リアサポは「学習者自身が気づく」。パラダイムが異なる
- **学術的裏付け**：CACM、CHI 2025、PRIMM研究など、教育研究のコンセンサスと合致する設計思想。「思いつき」ではなく研究に基づいたアプローチ

### 2. 再現性

- Webアプリとして誰でもアクセス可能な形態
- デモシナリオを1本通して動く状態を最優先で確保

### 3. Google技術の活用

- **Gemini API**：プロダクトの心臓部。パーソナライズ説明生成（Step 2）、コード生成（Step 3）、概念↔コードのマッピング（Step 4）、理解度判定（Step 5）の全ステップで使用
- **MCPサーバー**：Cursor/Claude Code等の開発環境と統合。成長に合わせて入口が変わる設計
- **なぜGoogleか**：Gemini APIが全ステップを一貫して担うことで、パーソナライズ→コード生成→マッピング→判定のパイプラインがシームレスに連携。構造化出力モードにより安定したJSON出力を実現

## 7. 着想の背景

### MiroFishからの着想

MiroFish（マルチエージェントAI予測エンジン）の設計思想「静的な情報を動的にモデリングし、相互作用から創発的な洞察を得る」から着想。

MiroFishは社会のエージェント間の相互作用から世論の創発を予測する。この考え方を学習支援に転用し、「学習者の概念間の依存関係を構造化し、理解の波及効果を可視化する」というアイデアに発展。

ただし学習の基礎概念の依存構造は「発見すべき未知」ではなく「既知の事実」であるため、エージェントシミュレーションではなく、グラフベースの軽量計算を採用。

### Antigravityからの進化

前身プロジェクトAntigravityは「AI待ち時間に能動学習チェックポイントを挟む」アプローチだった。リアサポはこれを進化させ、「AIがコードを書く前に介入する」ことで、待ち時間自体を予習の答え合わせに変換した。
