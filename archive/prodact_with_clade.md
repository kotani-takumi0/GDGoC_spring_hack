## テーマアンサー（確定）

### 核心の発見
Hello Worldはもう「最初の一歩」として機能していない。

### 構造の変化
**旧時代：**
- Hello World = 初心者が最初にやること
- `print("Hello World")` → 1行 → 書いた本人が理解できる
- 出力 = 理解（ギャップなし）
- だからHello Worldは「最初の一歩」として成立していた

**AI時代：**
- 初心者がAIに「アプリ作って」と言う → 数百行のコードが返る → 動く → でも理解できない
- 出力 ≠ 理解（巨大なギャップが生まれた）
- しかもそのギャップに、本人が気づけない（わからないことがわからない）
- Hello Worldが「最初の一歩」として機能しなくなった

---

## プロダクト概要

### コンセプト
AIが生成したコードの「構造マップ」をリアルタイムで構築し、自動診断によって「わからないことがわからない」を炙り出すツール。

### コア体験
**構造マップ × 自動診断**

1. **リアルタイム構築** — Kiro + Claude Codeで開発中、MCP接続を通じてアプリにイベントが飛び、構造マップにノードがリアルタイムで増えていく
2. **自動質問生成** — 各ノードに対してGemini APIが質問を自動生成（例：「このコンポーネントの役割は何？」「この引数は何のため？」）
3. **理解度の自動判定** — ユーザーが質問に答える → 答えられたら緑、曖昧なら黄色、答えられなかったら赤
4. **「わからない」の発見** — 自分では気づけなかった理解のギャップが構造マップ上に可視化される

### プロダクトが解く問題
- ✕ 「全部を理解しろ」
- ✕ 「自分でわからない場所を見つけろ」
- ○ 「わからないことがわからない、をわからせる」

## 構造マップの設計

### 二層構造（確定）
- **第1層：ファイル単位** — プロジェクト全体の構造。App.jsx、TodoList.jsx、utils.js などがノードとして並ぶ
- **第2層：ファイル内部** — ノードをクリックするとファイル内の関数・コンポーネント・フックが展開される

Googleマップの比喩：日本地図（第1層）→ 東京をクリック → 区や駅が見える（第2層）。それ以上は掘らない。

### ノードの状態
- 🟢 **理解済み** — 質問に正しく答えられた
- 🟡 **あやしい** — 曖昧な回答だった
- 🔴 **わからない** — 答えられなかった

※ ステータスはユーザーの自己申告ではなく、Geminiの質問に対する回答によって自動判定される

## 技術アーキテクチャ

### 三層構成
| レイヤー | ツール | 役割 |
|---|---|---|
| 仕様・設計 | **Kiro** | 要件定義 → 設計 → タスク分解 |
| 実装 | **Claude Code** | タスクを順に実装（MCPでアプリに通知） |
| 理解支援 | **このアプリ（MCPサーバー）** | 構造マップ構築・自動診断・ドキュメント生成 |

### 技術スタック
- **フロント**: React + TypeScript + React Flow（構造マップ描画）
- **AI**: Gemini API（構造解析・質問生成・回答判定）
- **リアルタイム通信**: MCP接続（Claude Code → アプリ）
- **ホスティング**: Firebase Hosting + Cloud Functions
- **学習連携**: NotebookLM用の学習ドキュメント生成（エクスポート機能）

### MCP接続の流れ
```
Claude Code（MCPクライアント）
  → タスク実行のたびにMCPサーバーのツールを呼ぶ
    → アプリ（MCPサーバー）がイベントを受け取る
      → WebSocket等でフロントに通知
        → 構造マップにノードが追加される
```

## NotebookLM連携
- アプリ内で「わからない」と判定されたノードについて、学習用ドキュメントを自動生成
- そのドキュメントをNotebookLMに渡すことで、音声解説・Q&A・深掘り学習が可能
- アプリ内の自動診断で「発見」→ NotebookLMで「深く学ぶ」という役割分担
- Google技術の活用アピールにもなる
