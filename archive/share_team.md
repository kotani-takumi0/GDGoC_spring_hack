## テーマアンサー

**Hello Worldはもう「最初の一歩」として機能していない。**

- 旧時代: `print("Hello World")` → 1行 → 書いて、動いて、理解できた → 出力 = 理解
- AI時代: 「アプリ作って」→ 数百行 → 動く → でも理解できない → 出力 ≠ 理解
- このギャップは初心者ほど大きく、しかも本人が気づけない

**Brand New Hello World = AI時代のエンジニアとしての最初の一歩**
→ 構造マップ上で最初のノードが🟢緑に変わる瞬間がそれ。

## プロダクト概要

AIが生成したコードの「構造マップ」をリアルタイムで構築し、「壊して予想」で理解度を可視化するツール。

**コアコンセプト**: AIがコードを書いている間の待ち時間を「理解」に使う生産性ツール。

**フレーミング**: 「AI時代にエンジニアとして成長する場所」。楽しいからやるアプリではなく、「やらなきゃいけないことを、今やるのが一番効率的」な場所を作る装置。

**ターゲット**: AI時代にエンジニアを志す初学者。

なぜこのターゲットか:
1. **チームでの説明責任**: エンジニアとして働く以上、コードの役割を説明する場面は避けられない
2. **プロンプト精度のための理解**: 「認証機能を作って」と「JWTで実装して」の差はなくならない。抽象度を下げた指示には構造的理解が必要
3. **初学者 = AI利用者**: 2026年にプログラミング学習を始めてAIを使わない人はいない。ターゲットは十分に広い

## ノードの状態遷移（6段階）

| 表示 | 意味 |
|------|------|
| ⬜ グレー（静止） | task.mdに記載済み、未実装 |
| ✨ グレー（点滅） | AIが今まさにこのファイルを書いている |
| ⬜ 白（静止） | 実装済み、理解チェック待ち |
| ✨ 白（点滅） | ユーザーが理解チェックに取り組んでいる |
| 🟢 緑 | 予想が当たった（理解している） |
| 🟡 黄 | 惜しい（部分的に理解） |
| 🔴 赤 | 外れた（理解できていない） |

## 理解チェックの手法

### ①事前学習（コードが書かれる前）
- task.mdの「役割」情報を元に、ファイル間の関係を予想する問題
- コードの話ではなく「役割」の話。初心者でもわかる言葉
- 正解のたびにエッジがアニメーションで光り、構造マップに意味が生まれる

### ②壊して予想（コードが書かれた後）
- 「この部分を消したらアプリはどうなる？」の選択式
- 予想 → 結果確認のサイクルで学習効果が高い
- 予想が当たった = 理解している、の明確な測定

### なぜクイズではなく「壊して予想」か
- クイズは受動的。「壊して予想」は能動的
- 予測 → 結果の確認は教育研究で学習効果が実証されている
- デモで「壊す」のは観客にわかりやすい
- ChatGPTに「問題出して」と聞くのとの差別化になる

## 操作的透明性: AIの思考をリアルタイムで見せる

### 最大の差別化ポイント
- Windsurf Codemaps → コードが書かれた**後**の静的な構造
- ChatGPT → コードが書かれた**後**に説明
- **このプロダクト → AIがコードを書いている過程そのものを構造マップ上でリアルタイムに見せる**

### 3つ目のMCPツール: report_progress
AIの実装中に思考ステップをリアルタイムで送信させる。

```
🤖 AIの現在地
  TodoList.jsx を書いています
  💭 「TodoItemをmapで繰り返し表示する構造にします。
      forEachではなくmapを選んだ理由: Reactでは戻り値が必要だから」
  ステップ: ████████░░ 3/5
```

**AIを遅くするのではなく、AIの独り言が多くなるだけ。** その独り言がユーザーの学習時間になる。

## 技術スタック

### 全体構成

```
Google Antigravity（IDE）
  → Gemini 3 がコードを書く
    → MCP接続（ネイティブサポート）
      → 理解支援アプリ（自分たちが作るもの = MCPサーバー）
```

### 自分たちが作るもの

| レイヤー | 技術 | 役割 |
|---|---|---|
| フロント | React + TypeScript | アプリUI |
| 構造マップ（左） | ツリーコンポーネント | ファイル一覧 + 理解度カラー |
| 構造マップ（右） | React Flow | 選択ファイルの内部構造グラフ |
| MCPサーバー | Node.js | Antigravityからのイベント受信 |
| リアルタイム通知 | WebSocket | MCPサーバー → フロントへの通知 |
| ホスティング | Firebase Hosting + Cloud Functions | デプロイ・サーバーレス |
| 状態管理 | Firestore | 構造マップのリアルタイム同期 |

### MCP接続の設計

**設定ファイル**: `~/.gemini/antigravity/mcp_config.json`

**MCPツール（3つ）**:

| ツール | タイミング | 送るデータ |
|--------|-----------|-----------|
| `report_task_plan` | タスク分解直後（1回） | タスク一覧、ファイル構成、依存関係、各ファイルの目的 |
| `report_progress` | AI実装中の各ステップ（複数回） | ファイル名、ステップ説明、判断理由、進捗 |
| `report_file_change` | ファイル完成時（ファイルごとに1回） | ファイルパス、コード内容、変更差分 |

**エージェントルール（3つ）**: `.agent/rules/` に配置、全て `always_on`

### Google技術パイプライン

```
report_task_plan（タスク分解時）
  → 構造マップの骨格を描画
  → Gemini Flash: 事前学習の問題を生成

report_progress（AI実装中）
  → 「AIの現在地」パネルにリアルタイム表示

report_file_change（ファイル完成時）
  → Gemma 3（Cloud Functions）: 高速な構造解析
    → Gemini Flash（Vertex AI経由）: 壊して予想の問題を生成
      → ユーザーが回答
        → Gemini Pro（Vertex AI経由）: 回答判定
          → 赤ノード → NotebookLM: 音声解説の自動生成
```

| Google技術 | 役割 | なぜこれか |
|---|---|---|
| Antigravity | IDE・コード生成 | Gemini 3搭載、MCP対応のGoogle製IDE |
| Gemma 3 | リアルタイム構造解析 | 軽量・高速、Cloud Functionsで稼働 |
| Gemini Flash | 問題生成（事前学習 + 壊して予想） | 速度重視 |
| Gemini Pro | 回答判定 | 精度重視の意味理解 |
| Vertex AI | モデル統合管理 | Gemma/Geminiを同一基盤で管理 |
| NotebookLM | 深掘り学習 | 赤ノードの音声解説を自動生成 |
| Firebase | インフラ | Hosting / Cloud Functions / Firestore |

## NotebookLM連携（検証済み）

### 実現方法
- `notebooklm-mcp-cli`（jacob-bd/notebooklm-mcp-cli）を使用
- **注意**: 非公式API依存、認証はブラウザCookie抽出方式。デモ当日に壊れるリスクあり → フォールバック必須

### 自動化フロー
1. 赤ノードが出る → バックエンドが `notebook_create` で専用ノートブック作成
2. 該当コード・文脈を `source_add` で追加
3. `studio_create` で音声解説を自動生成
4. ユーザーは他のノードの理解チェック中に耳で学べる

→ NotebookLMが「おまけ」ではなく「パイプラインの一部として自動化される」形

## 削ぎ落としの方針

### 基本原則
**「デモの5分間に登場しないものは全部削る」**

### デモに出る = 必ず実装するもの
- MCP連携（report_task_plan, report_file_change）でノード自動生成
- 構造マップUI（ハイブリッド型: 左ツリー + 右React Flow）
- 理解チェック（壊して予想・選択式）
- ノードの色変化（グレー → 白 → 緑/黄/赤）
- 「Hello, World.」表示

### 余裕があれば実装
- report_progress（AIの思考リアルタイム表示）
- NotebookLM音声解説の自動生成
- 点滅アニメーション（グレー点滅・白点滅）
- 事前学習フェーズ

### 削る or 口頭で触れるだけ
- Gemma構造解析（Flashで代替可能なら統合）
- 事前学習フェーズ（デモでは省略、口頭で言及）

### デモ形式の推奨
**ハイブリッド**: MCP連携→ノード出現は録画、理解チェック→緑に変わるところはライブ。MCP当日死亡リスクを回避しつつライブ感も確保。

## 既知の弱点と対策

| 弱点 | 対策・回答 |
|------|-----------|
| AIが速すぎて並走が成立しない | デモ題材を調整 / 「書き終わった直後でも価値がある」と言語化 |
| 事前学習の必要性が伝わりにくい | デモでは省略して口頭で触れるだけにする |
| 壊して予想がMVPだとただのテキスト | ビジュアル比較をどこまで作り込むか要判断 |
| Gemmaの必要性が弱い | 「Flashでよくない？」への回答を用意する |
| 「待ち時間がなくなったら？」 | AIが速くなるほど出力と理解のギャップは広がる。理解のタイミング提供の価値は変わらない |
| NotebookLMが当日壊れるリスク | 非公式API依存のためフォールバック必須 |
| MCPエージェントルールの信頼性 | LLMへのプロンプトであり非決定的。デモはハイブリッド（録画+ライブ）で回避 |
