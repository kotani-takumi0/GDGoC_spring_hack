## 1. テーマアンサーの精錬

### 元の構想
「Hello Worldはもう最初の一歩として機能していない」→ 構造マップで診断するツール

### 議論で得たフレーミング転換

**旧フレーミング**: 診断ツール（「わからないことを見つける」）
**新フレーミング**: 生産性ツール（「AIが書いてる間の待ち時間を理解に使う」）

根拠:
- これからの時代、コードを書くことはなくなり、理解することの価値が上がる
- AIを遅くしてまで理解を挟む（ペアコーディング型）は逆方向
- AIが全速力で書く → その間に人間が理解を進める → 出力と理解が並走する

### テーマとの接続
- 旧Hello World: 出力と理解が同時だった
- AI時代: 出力は一瞬、理解は置き去り
- Brand New Hello World: 出力と理解を再び並走させる

### プロダクト体験のテーマ直結
- 構造マップ上で最初のノードが緑に変わる瞬間 = AI時代の新しいHello World
- AIがまだコードを書いている最中に、人間が最初の理解を得る → その瞬間がHello World

## 2. プレゼン構成の改善案

### 締めの改善
**元**: 「わからないことがわからない、をわからせる。これが僕たちのBrand New Hello Worldです。」

**改善案**:
「昔のHello Worldは、書いて、動いて、わかった。1行で。」
「僕たちのHello Worldは、問われて、答えて、わかった。1ノードで。」
「最初のノードが緑に変わる。それがAI時代の Brand New "Hello World." です。」

### 演出案
- 最初のノードが緑に変わったとき、アプリが「Hello, World.」と表示する
- デモのクライマックスとテーマアンサーとプロダクト体験が同時に着地する

## 4. 構造マップUI: ハイブリッド型に決定

### 元の設計
React Flow のノードグラフのみ（二層構造）

### 問題点
- ノード数が増えると視認性が急激に落ちる
- 2D自由配置は情報の優先度を表現しにくい
- リアルタイム追加時にレイアウトが再配置されて混乱する

### 検討した代案
1. ディレクトリツリー型（VS Code風）→ デモ映えしない
2. 同心円型 → 循環依存で破綻
3. **ツリー + ノードグラフのハイブリッド** ← 採用
4. ヒートマップ型 → 関係性が見えない

### 採用: 案3 ハイブリッド型

```
┌─────────────┬──────────────────────────┐
│ 📁 src/     │                          │
│  📄 App  🟢 │   [useState] 🟢          │
│  📄 Todo 🔴 │       ↓                  │
│  📄 Item 🟡 │   [handleAdd] 🔴→[Todo]  │
│  📄 utils🟢 │       ↓                  │
│             │   [return JSX] 🟡        │
└─────────────┴──────────────────────────┘
  左: ツリー       右: ノードグラフ
  (ナビゲーション)  (選択ファイルの内部構造)
```

**左パネル: ディレクトリツリー**
- ファイル一覧 + 理解度カラー
- スクロール可能 → 100ファイルでもスケール
- ナビゲーション用

**右パネル: React Flow ノードグラフ**
- 左で選択したファイルの内部構造を表示
- 関数・コンポーネント・フックがノード
- 依存関係がエッジ
- 1ファイルあたり5-15ノード → React Flowが破綻しない範囲

**採用理由**:
- デモではリアルタイムでノード追加される右パネルが映える
- 実用では左のツリーがスケールを支える
- 第2層のノード数が制御可能な範囲に収まる

## 6. MCP接続の具体設計（確定）

### 元の不備
「いつ・何をトリガーに・どんなデータを送るか」が全て未定義だった。

### Antigravity の MCP 仕様で解決

**設定ファイル**: `~/.gemini/antigravity/mcp_config.json`

```json
{
  "mcpServers": {
    "understanding-app": {
      "command": "node",
      "args": ["/path/to/your-mcp-server/index.js"],
      "env": {
        "MCP_MODE": "stdio"
      }
    }
  }
}
```

### エージェントルールの起動モード

| モード | 挙動 | 用途 |
|--------|------|------|
| `always_on` | 常に自動適用 | ← これを使う |
| `manual` | プロンプトで言及した時だけ | - |
| `model_decision` | AIが必要と判断した時 | - |
| `glob` | 特定ファイルパターンにマッチした時 | - |

### MCPツール定義（2つ）

| ツール | タイミング | 送るデータ |
|--------|-----------|-----------|
| `report_task_plan` | タスク分解直後（1回） | タスク一覧、ファイル構成、依存関係、各ファイルの目的 |
| `report_file_change` | ファイル作成/編集ごと | ファイルパス、コード内容、変更差分 |

### エージェントルール（2つ）

**`.agent/rules/understanding-plan.md`**
```markdown
---
activation: always_on
---

タスク分解が完了したら、必ず understanding-app の
report_task_plan ツールを呼び出し、
タスク一覧・ファイル構成・依存関係・各ファイルの目的を送信すること。
```

**`.agent/rules/understanding-code.md`**
```markdown
---
activation: always_on
---

ファイルを作成または編集したら、必ず understanding-app の
report_file_change ツールを呼び出し、ファイルパスと内容を送信すること。
```

### ノードの状態遷移（4段階 + 2状態）

```
⬜ グレー: タスク分解で出現（未実装）
⬜ 白:     コード実装済み（未理解・チェック問題生成済み）
🟢 緑:     理解チェック通過
🟡 黄:     あやしい
🔴 赤:     理解できていない
```

ユーザーは構造マップを一目見るだけで:
- **グレーが多い** → AIがまだ書いてる最中
- **白が多い** → 書き終わったけど理解が追いついてない
- **緑が多い** → 順調に理解が進んでる

### 確定したワークフロー（全体フロー）

```
① ユーザー: 「Todoアプリを作って」
   │
   ▼
② Gemini 3 がタスク分解（Antigravityの標準動作）
   │  例:
   │  - Task 1: App.jsx（エントリポイント）
   │  - Task 2: TodoList.jsx（リスト表示）
   │  - Task 3: TodoItem.jsx（個別アイテム）
   │  - Task 4: useTodos.js（状態管理フック）
   │  - Task 5: storage.js（ローカルストレージ）
   │
   ├─── MCP: report_task_plan（自動発火）───┐
   │                                        ▼
   │                              理解支援アプリが受け取る
   │                                → 全体の骨格を構造マップに描画
   │                                → 全ノードがグレー（未実装）で出現
   │                                → ファイル間の依存関係もこの時点で見える
   │
   ▼
③ Gemini 3 が Task 1 から実装開始
   │
   ├─── MCP: report_file_change（自動発火）───┐
   │                                           ▼
   │                                 グレー → 白（実装済み・未理解）に変わる
   │                                   → Gemma が構造解析 → 内部ノード出現
   │                                   → Gemini Flash が理解チェック生成
   │
   ▼
④ ユーザーが待ち時間に理解チェック開始
   │  全体像が見えている状態で
   │  「今AIはここを書いてる」「自分はここを理解してる」が一目でわかる
   │
   ├─ 最初のノードが緑に → 🎉「Hello, World.」
   │
   ▼
⑤ Gemini 3 が Task 2, 3... と進む
   │  グレー → 白 → （ユーザーが理解） → 緑/黄/赤
   │
   ▼
⑥ 全タスク完了時
   │  構造マップ: 緑と赤が混在
   │  赤 → NotebookLM で深掘り
   │
   ▼
⑦ 全ノード緑 → プロジェクト全体を理解した状態
```


### report_task_plan の効果
- **理解にコンテキストが生まれる**: 個別ファイルだけでなく「全体の中でこのファイルが何をしているか」がわかる
- **進捗が直感的**: グレー→白→緑の遷移で、AIの進捗と自分の理解の進捗が同時に見える
- **デモが映える**: タスク分解した瞬間に骨格がバッと出る → 徐々に色が変わっていく

## 7. Google技術の活用設計（確定）

### パイプライン全体

```
コードイベント発生（ファイル作成/編集）
  ↓
① Gemma 3（Cloud Functions）— 高速な一次構造解析
  ↓
② Gemini Flash（Vertex AI経由）— 理解チェック問題の生成
  ↓
③ ユーザーが回答 → Gemini Pro（Vertex AI経由）— 回答判定
  ↓
④ 赤ノード → NotebookLM — 音声解説を自動生成
```

### 各技術の役割と「なぜそれか」

**Gemma 3（Cloud Functions上）**: リアルタイム構造解析
- ファイル作成/編集ごとに関数・コンポーネント・フック・依存関係を抽出
- リアルタイム性が必要 → Gemini APIだとレイテンシが大きすぎる
- Gemma 3は軽量で数百ミリ秒で返せる
- 構造解析は定型的タスクなので小さいモデルで十分

**Gemini Flash（Vertex AI経由）**: 理解チェック生成
- 各ノードの役割を問う質問を生成
- 速度重視のタスク → Flash

**Gemini Pro（Vertex AI経由）**: 回答判定
- ユーザーの回答を受け取り理解度を判定（緑/黄/赤）
- 文脈を踏まえた意味理解が必要 → Pro

**Vertex AI**: 統合管理基盤
- Gemma・Geminiを同一基盤で管理
- プロンプト管理・ログ・モニタリングを統一
- 「本番運用を意識した構成」として審査にアピール

**NotebookLM**: 赤ノードの深掘り学習
- 赤ノードのコード・文脈・質問を自動でソースとして追加
- 音声解説（Audio Overview）を自動生成
- 他ノードの理解チェック中に耳で学べる → 「待ち時間活用」思想と一致

**Firebase**: インフラ
- Hosting → Webアプリ配信
- Cloud Functions → Gemma稼働、APIプロキシ
- Firestore → 構造マップの状態リアルタイム同期

### 審査への語り方
> 「Antigravityでコードを書き、Gemmaが即座に構造を解析し、Geminiが理解チェックを生成し、
> 理解できなかった部分はNotebookLMが音声で解説する。全てVertex AI上で統合管理しています。
> Googleのエコシステム内で完結するプロダクトです。」

## 8. 競合調査 & 理解チェック手法の転換（2026-02-24）

### 競合調査: Windsurf Codemaps

Windsurf Codemaps（Cognition社、2025年9月ベータ、11月公開）が最も近い競合。

**Codemapsの特徴**:
- オンデマンド・プロンプト駆動で構造マップを生成（リアルタイムではない）
- ノード + エッジで依存関係・実行フローを可視化（Mermaid形式）
- フローチャート、シーケンス図、クラス図を生成可能
- Cascade（AIエージェント）に `@{codemap}` でコンテキストとして渡せる
- スローガン: 「Understand Code Before You Vibe It」「Fight slop」

**Codemapsのターゲット**: プロのエンジニアが新しいチーム/プロジェクトに入った時のコード理解
**自分たちのターゲット**: AIで初めてモノを作れた人（バイブコーダー）がコードを理解したい

**Codemapsにないもの**:
- 理解度の測定・可視化（緑/黄/赤）
- 理解チェック（クイズや破壊シナリオ）
- リアルタイム生成（AIがコードを書いている最中の連携）
- 事前学習（コードが書かれる前の概念教育）
- 音声解説

### 課題の認識

元の設計（構造マップ + クイズ）では差別化が弱い:
- 構造マップ → Codemapsの劣化版になりかねない
- クイズ → ChatGPTに「問題出して」で済む
- 「既存ツールの機能をつなぎ合わせた」だけに見えるリスク

### 理解チェック手法の転換: クイズ → 「壊して予想する」

**元の手法**: 自由記述 or 選択式クイズ（「この関数の役割は？」）
**新しい手法**: 破壊シナリオの予想（「この部分を消したらアプリはどうなる？」）

**なぜ変えるか**:
- クイズは受動的。「壊して予想」は能動的
- 予想 → 結果確認のサイクルは学習効果が高い（既存の教育研究でも実証済み）
- デモで「壊す」のは観客にわかりやすい
- 理解度の測定が明確（予想が当たった = 理解している）

### task.md による構造マップ起点の設計（2026-02-24）

#### 背景
Kiroのtask.md（タスクファイル）のように、AIがタスク分解した結果をファイルとして出力させる。
このファイルが構造マップの情報源となり、事前学習・ノード生成の起点になる。

#### task.md の形式（例）

```markdown
# task.md
## Task 1: App.jsx（エントリポイント）
- 役割: アプリ全体のルート。TodoListを配置
- 依存: TodoList.jsx, useTodos.js
- 状態: [ ] 未着手

## Task 2: TodoList.jsx（リスト表示）
- 役割: Todoの一覧を表示
- 依存: TodoItem.jsx
- 状態: [ ] 未着手

## Task 3: TodoItem.jsx（個別アイテム）
- 役割: 1つのTodoの表示と操作（完了・削除）
- 依存: なし
- 状態: [ ] 未着手

## Task 4: useTodos.js（状態管理フック）
- 役割: Todoの追加・削除・完了の状態管理
- 依存: storage.js
- 状態: [ ] 未着手

## Task 5: storage.js（ローカルストレージ）
- 役割: ブラウザを閉じてもデータを保持
- 依存: なし
- 状態: [ ] 未着手
```

#### task.md を起点にした流れ

1. task.md の「役割」「依存」をパースして構造マップのノードとエッジを生成
2. task.md の「役割」テキストを元に Gemini Flash が事前学習の問題を生成
3. task.md の「状態」がAIの実装進捗と連動（MCP経由で更新）

#### なぜ task.md を挟むか

- **情報源が明確**: AIの曖昧な出力をパースする必要がなく、構造化されたデータから確実にノードを作れる
- **事前学習の精度**: 各ファイルの「役割」がテキストで書かれているので、問題生成が安定する
- **Kiroとの親和性**: 既存ツールユーザーに馴染みのある形式
- **デバッグしやすい**: task.md を見ればMCPで何が送られたか一目でわかる

### ノードのインタラクティブ状態（2026-02-24）

構造マップのノードをAIの実行と連動させ、ライブダッシュボード化する。

#### 状態一覧（点滅を追加）

```
⬜ グレー（静止）    : task.mdに記載済み、未実装
✨ グレー（点滅）    : AIが今まさにこのファイルを書いている
⬜ 白（静止）        : 実装済み、理解チェック待ち
✨ 白（点滅）        : ユーザーが理解チェックに取り組んでいる
🟢 緑              : 予想が当たった（理解している）
🟡 黄              : 惜しい（部分的に理解）
🔴 赤              : 外れた（理解できていない）
```

#### 点滅の実装トリガー

| 状態変化 | トリガー |
|---------|---------|
| グレー静止 → グレー点滅 | MCP: report_file_change の開始通知を受信 |
| グレー点滅 → 白静止 | MCP: report_file_change の完了通知を受信 |
| 白静止 → 白点滅 | ユーザーがそのノードの理解チェックを開始 |
| 白点滅 → 緑/黄/赤 | ユーザーが予想を送信し、判定が返る |

#### この変更の効果

- **「静的な地図」→「ライブダッシュボード」**: 構造マップが生きている感覚になる
- **AIと自分の進捗が並走して見える**: 「AIは今ここ、自分はここまで理解した」が一目でわかる
- **デモ映え**: 点滅→白→緑とノードの色が次々変わっていく視覚的なインパクト

### 新ワークフロー（確定版 v2）

```
① ユーザー:「Todoアプリを作って」（Antigravity上で指示）
   │
   ▼
② Gemini 3 がタスク分解 → task.md を出力
   │
   ├─── MCP: report_task_plan（task.mdの内容を送信）───┐
   │                                                    ▼
   │                              task.mdをパースして:
   │                              → 全ノードをグレー（静止）で構造マップに描画
   │                              → 「依存」フィールドからエッジを描画
   │                              → 「役割」フィールドから事前学習の問題を生成
   │
   ▼
③ 【事前学習: 予想フェーズ】（コードが書かれる前）
```

## 9. 操作的透明性: AIの思考をリアルタイムで見せる（2026-02-24）

### 背景

市場調査の結果、「操作的透明性（Operational Transparency）」がプロダクトの核心機能になり得ると判断。

**操作的透明性とは**: システムが何をしているかをユーザーに見せること。研究では、AIが何をしているか見せると待ち時間の満足度・信頼度が上がることが実証されている。

### なぜこれがプロダクトの核心になるか

- Windsurf Codemaps → コードが書かれた**後**の静的な構造を見せる
- ChatGPT → コードが書かれた**後**に説明する
- **このプロダクト → AIがコードを書いている過程そのものを構造マップ上でリアルタイムに見せる**

**AIの思考過程をリアルタイムで構造マップに表示するツールは存在しない。** これが最大の差別化。

### 3つ目のMCPツール: report_progress

既存の2ツールでは、ファイル実装中の情報が取れない（点滅するだけ）。
3つ目のツールを追加し、AIの思考ステップをリアルタイムで送信させる。

| ツール | タイミング | 粒度 |
|--------|-----------|------|
| `report_task_plan` | タスク分解直後（1回） | プロジェクト全体 |
| `report_progress`（NEW） | AIが判断・実装の各ステップを進めるたびに | ステップ単位 |
| `report_file_change` | ファイル完成時 | ファイル単位 |

#### report_progress のデータ

```json
{
  "file": "TodoList.jsx",
  "step": "リスト表示の実装方針を決定",
  "detail": "mapを使ってTodoItemを繰り返し表示する構造にします",
  "reasoning": "forEachではなくmapを選択。理由: Reactでは戻り値が必要だから",
  "step_number": 3,
  "total_steps": 5
}
```

#### エージェントルール

```markdown
# .agent/rules/understanding-progress.md
---
activation: always_on
---

ファイルの実装中、以下のタイミングでunderstanding-appの
report_progressツールを呼び出すこと：
- 実装方針を決定したとき
- 主要な関数/コンポーネントの定義を開始したとき
- 重要な技術的判断（ライブラリ選択、パターン選択）をしたとき

送信内容: 現在のファイル名、ステップの説明、判断の理由
```

### UIイメージ: AIの現在地パネル

```
┌────────┬──────────────────────────────────────┐
│ 📁 src │                                      │
│ App 🟢 │  [useState]🟢                       │
│ Todo✨  │      ↓                              │
│ Item⬜  │  [handleAdd]🟡→ [TodoList] ✨        │
│ hooks⬜ │      ↓                              │
│ store⬜ │  [return JSX]🟢                      │
│        │                                      │
│        │  ┌──────────────────────────────────┐ │
│        │  │ 🤖 AIの現在地                     │ │
│        │  │                                  │ │
│        │  │ TodoList.jsx を書いています        │ │
│        │  │                                  │ │
│        │  │ 💭 「TodoItemをmapで繰り返し      │ │
│        │  │    表示する構造にします。          │ │
│        │  │    forEachではなくmapを選んだ理由: │ │
│        │  │    Reactでは戻り値が必要だから」   │ │
│        │  │                                  │ │
│        │  │ ステップ: ████████░░ 3/5          │ │
│        │  └──────────────────────────────────┘ │
└────────┴──────────────────────────────────────┘
```

### 予想との統合フロー

```
① AIがTodoList.jsxを書き始める → ✨点滅

② 「AIの現在地」パネルに思考が流れ始める
   「TodoList.jsxを書いています」
   「まず、Todoの配列をどう表示するか決めます...」

③ この瞬間に予想問題が出る（Twitchの予測モデル）
   「AIはリスト表示にどの方法を使うと思いますか？」
   [A] map  [B] forEach  [C] for文

④ ユーザーが予想する

⑤ AIの思考が進む（リアルタイムで見える）
   「mapを選択しました。理由: Reactではreturnが必要...」

⑥ ユーザーの予想と照合
   → 当たった:「AIと同じ判断ができました！」
   → 外れた:「mapを選ぶ理由は〜」

⑦ AIがファイルを書き終わる → 白に変わる
⑧ 壊して予想フェーズに移行
```

### ペーシング: 「遅くする」のではなく「共有させる」

report_progressでAIの思考ステップを細かく報告させることで、
AIは全力で書いているが、思考の共有（report_progress呼び出し）が自然な間を作る。

**AIを遅くするのではなく、AIの独り言が多くなるだけ。**
その独り言がユーザーの学習時間になる。

これにより、以前のブレストで否定した「AIを遅くしてまで理解を挟む」ことなく、
ユーザーが予想・学習する時間が自然に確保される。

### 効果まとめ

| 観点 | 効果 |
|---|---|
| 死んでる時間 | ゼロになる。AIの思考がリアルタイムで流れ続ける |
| ライブ感 | ライブコーディング配信を見てるのと同じ体験 |
| 学習効果 | 「なぜmapを使うか」をAIの判断理由として見る → 教科書より文脈がある |
| 予想との統合 | AIの次の判断を予想 → 結果がリアルタイムで見える → 予測エラー効果で記憶定着 |
| 差別化 | AIの思考過程をリアルタイムで構造マップに表示するツールは世界に存在しない |

## 10. プロダクトのフレーミング転換: 「楽しいからやる」ではなく「やるべきことを今やる場所」（2026-02-24）

### 批判と反論の経緯

**批判**: ターゲット（バイブコーダー）はAIがコードを書いている間、スマホを見る。学習アプリなんて開かない。

**反論（確定）**: このプロダクトはスマホより面白い必要はない。

### 核心の洞察

コードを理解することは **「いつかやらなければいけないこと」** である。

- エンジニア志望の学生 → 面接で「使った技術を説明して」と聞かれる
- ハッカソン参加者 → 審査員にプロダクトの技術を説明する必要がある
- プロダクトを改善したい人 → コードを理解しないと変更できない

**やらなきゃいけない。でも「いつやるか」を決められない。** 課題と同じ。

### このプロダクトの役割

**「コード理解」という、いつかやらなければいけないタスクに対して、最適なタイミング（AIが書いている間）と環境（構造マップ + 予想チェック）を強制的に作る装置。**

- スマホのほうが楽しい。でもやる。やらなきゃいけないから。
- 課題だって楽しくない。でもやる。期限があるから。
- このプロダクトは「今やるのが一番効率的」なタイミングを提示する。

### Duolingoとの根本的な違い

| | Duolingo | このプロダクト |
|---|---|---|
| 動機の源泉 | 楽しいからやる（内発的 + ゲーミフィケーション） | やらなきゃいけないからやる（外発的 + 実用的必要性） |
| 引き留め方 | ストリーク、XP、リーグ（エンタメ要素） | 最適なタイミングと環境の提供（利便性） |
| 競合 | Netflix、TikTok（娯楽と戦う） | 「後でやろう」（先延ばしと戦う） |
| 失敗モード | 飽きてやめる | そもそも開かない |

### 設計への示唆

- ゲーミフィケーション（ストリーク、ポイント、競争）は必須ではない
- 代わりに必要なのは:
  1. **「今やるのが一番効率的」と納得させるオンボーディング**
  2. **やり始めたら摩擦なく進められるUX**
  3. **やった結果が明確に見える成果物**（面接で使える理解度マップ等）
- 「楽しさ」ではなく「やりやすさ」と「やった実感」で設計する

### プレゼンとの接続

導入:
> 「皆さんもこの1週間でAI使いましたよね？」
> 「そのAIが書いたコード、全部説明できますか？」

→ 聴衆が「やばい、説明できない」と思う
→ 「僕たちのプロダクトは、AIがコードを書いているまさにその時間に、理解する場所を作ります」

テーマアンサーとプロダクトのフレーミングが一直線につながる。
