# Google Stitch用 UIモックプロンプト

各画面ごとにStitchに入力するプロンプトをまとめる。

---

## 画面1: Step 1 — シナリオ・経験レベル選択

```
プログラミング学習支援Webアプリのトップ画面を作ってください。

画面の目的：学習者がシナリオと自分の経験レベルを選ぶ

構成要素：
- ヘッダー：アプリ名「リアサポ」とキャッチコピー「AI時代の新しい"Hello World"体験」
- シナリオ選択エリア：カード形式で選択肢を並べる
  - Todoアプリ（選択可能、メインシナリオ）
  - 他のシナリオはグレーアウトで「Coming Soon」表示
- 経験レベル選択エリア：3つの選択肢をラジオボタンまたはカード形式で表示
  - 「完全初心者 — プログラミング自体が初めて」
  - 「Python経験あり — Pythonは書けるが、Web開発・Reactは初めて」
  - 「他言語でのWeb開発経験あり — Reactは初めて」
- 「ロードマップを見る」ボタン（両方選択後に有効化）

デザインの方向性：
- 清潔感のある白基調、アクセントカラーは青系
- 初学者が安心感を持てるシンプルなデザイン
- モバイル対応不要、デスクトップ画面のみ
```

## 画面2: Step 2 — 概念ロードマップ表示（実装前）

```
プログラミング学習支援Webアプリの「概念ロードマップ」画面を作ってください。

画面の目的：AIにコードを書かせる前に、学習者が「何を理解すべきか」の全体像を知る

構成要素：
- 画面中央にロードマップ（ノード＋矢印のフローチャート形式）
  - 7つの概念ノードを上から下に依存関係順で配置
  - 各ノードは角丸の長方形で、以下の2行を表示：
    - 1行目：技術用語（太字）例：「状態管理」
    - 2行目：平易な一言説明（小さめフォント）例：「データを覚えておく仕組み」
  - ノード間を矢印（→）で接続し、依存関係を表現
  - ノードの色はなし（グレーまたは白の統一色。まだ評価していない状態）
- 依存関係の構造：
  - 「コンポーネント設計」から「JSX記法」と「状態管理」に分岐
  - 「JSX記法」と「状態管理」が「リスト表示」に合流
  - 「状態管理」から「イベントハンドリング」と「永続化」にも分岐
  - 「リスト表示」から「条件付きレンダリング」に接続
- 下部に「OK、実装して」ボタン

デザインの方向性：
- ロードマップが地図のように見える、見通しの良いレイアウト
- ノードは十分な余白を持って配置
- 矢印は曲線で柔らかい印象に
```

## 画面3: Step 3 — コード生成中（ローディング）

```
プログラミング学習支援Webアプリの「コード生成中」画面を作ってください。

画面の目的：AIがコードを生成している間の待機画面

構成要素：
- 画面中央にローディングアニメーション
- 「AIがTodoアプリのコードを生成しています...」のテキスト
- Step 2で表示したロードマップを薄く背景に表示（学習者が待ち時間に眺められる）
- プログレスバーまたはステップインジケーター

デザインの方向性：
- 待ち時間を退屈に感じさせない工夫
- ロードマップが背景にあることで、待ちながら予習できる意味合い
```

## 画面4: Step 4 — ロードマップ×コード並列表示（実装後）

```
プログラミング学習支援Webアプリの「ロードマップとコードの並列表示」画面を作ってください。

画面の目的：概念ロードマップと生成されたコードを並べ、どの概念がコードのどこに対応するか視覚的に示す

構成要素：
- 左パネル（幅40%）：概念ロードマップ
  - Step 2と同じノード構造を縦に配置
  - 各ノードの下にコード例が追加表示されている
    例：「状態管理」ノードの下に「→ const [todos, setTodos] = useState([])」
  - ノードはクリック可能（クリックすると右パネルの対応箇所がハイライト）
- 右パネル（幅60%）：生成されたコード
  - シンタックスハイライト付きのコード表示
  - 概念ノードに対応する箇所が色付きの背景でハイライトされている
  - ハイライトの色は各ノードと対応（ノードの枠線と同じ色）
- 左右パネル間を線（カーブした接続線）で結ぶ
  - 各ノードから対応するコード箇所へ線が伸びている
  - ホバーすると該当する線とコード箇所が強調表示
- 下部に「理解度チェックに進む」ボタン

デザインの方向性：
- IDEのようなコード表示（ダークテーマ）と、ロードマップ（ライトテーマ）のコントラスト
- 接続線は普段は薄く表示し、ホバーで強調
- この画面がプロダクトの核心なので、最も印象的なビジュアルにする
```

## 画面5: Step 5 — 理解度評価

```
プログラミング学習支援Webアプリの「理解度チェック」画面を作ってください。

画面の目的：学習者が各概念について自分の言葉で説明し、理解度を確認する

構成要素：
- 左パネル：ロードマップ（Step 4と同じ構造）
  - 各ノードに理解度の色が付く（回答後）
    - 🟢 緑：説明できている
    - 🟡 黄：部分的に理解
    - 🔴 赤：理解できていない
    - グレー：未回答
  - 現在回答中のノードが枠線で強調
- 右パネル：回答エリア
  - 上部：現在の概念ノード名と対応するコード箇所の表示
    例：「状態管理 — const [todos, setTodos] = useState([])」
  - 中央：質問テキスト「このコードが何をしているか、自分の言葉で説明してください」
  - テキスト入力エリア（3〜5行程度）
  - 「回答する」ボタン
  - 回答後：AIの判定結果と一言フィードバックが表示される
- 下部：プログレス表示（7問中○問回答済み）

デザインの方向性：
- ロードマップのノードが回答ごとに色づいていく様子が見えるのが体験の核心
- テスト感を出しすぎない、穏やかなトーン
- 回答後のフィードバックは短く、励ましを含む
```

## 全体の画面遷移

```
Step 1（選択） → Step 2（ロードマップ） → Step 3（生成中） → Step 4（並列表示） → Step 5（理解度チェック）
```

ヘッダーにステップインジケーター（1 → 2 → 3 → 4 → 5）を常時表示し、現在位置を示す。
